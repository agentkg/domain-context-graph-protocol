# DCG Ingest Workflow

**Spec ID:** DCG-001-PW
**Status:** Draft
**Date:** 2026-06-26
**Implements:** [DCG-001](../rfc/01-rfc.md)

---

## 1. Abstract

This specification defines a reusable workflow for populating any DCG domain
knowledge graph. It covers confidence-tiered entity classification, graph state
management, a structured five-pass population protocol, draft subgraph isolation,
human review via ReviewItem entities, and expiry/cleanup rules.

DCG-PW-001 is protocol-level guidance — it applies regardless of the domain
(security, payments, infrastructure) or the tooling used to populate the graph.
Implementations MAY automate any or all passes, but the workflow structure and
confidence semantics defined here MUST be followed for interoperability.

---

## 2. Terminology

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD",
"SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be
interpreted as described in [RFC 2119](https://www.ietf.org/rfc/rfc2119.txt).

Additional definitions:

- **Population Run**: A structured sequence of passes that adds or updates
  entities and relations in a domain graph.
- **Confidence Tier**: A classification (T1–T4) indicating the trustworthiness
  of an entity's source data and the review level required before commit.
- **Draft Entity**: An entity stored in the draft subgraph (`graphs/draft.json`)
  pending human approval.
- **Committed Entity**: An entity in the main graph (`graphs/default.json`) that has
  passed validation and any required review.
- **ReviewItem**: An entity type (see PW-57) that flags another entity for
  human review. Implementations using this workflow SHOULD register
  `ReviewItem` in the project's ontology.
- **Promotion**: Moving a draft entity from the draft subgraph to the main graph
  after human approval.

## 3. Confidence Tiers

Entities entering a DCG domain graph are classified into four confidence tiers.
The tier determines where the entity is stored, what review is required, and what
attributes the entity must carry.

### 3.1 Tier Definitions

**PW-1.** Every entity added during a population run MUST be assigned exactly one
confidence tier (T1, T2, T3, or T4).

**PW-2.** The confidence tier MUST be recorded as a `confidence` attribute on the
entity, using the tier label as the string value:

| Tier | `confidence` value | Description |
|------|--------------------|-------------|
| T1   | `"official"`       | Authoritative, externally verified sources |
| T2   | `"moderate"`       | Reliable enrichment sources, spot-check review |
| T3   | `"draft"`          | Low-confidence or AI-generated, requires approval |
| T4   | `"dropped"`        | Rejected or expired, cleanup candidate |

**PW-3.** **T1 — Official.** Entities from authoritative, externally maintained
sources (e.g., MITRE CWE catalog, OWASP Top 10 lists, published RFCs, official
vendor documentation). T1 entities require no human review and are committed
directly to the main graph.

**PW-4.** T1 entities MUST include a `source_url` attribute pointing to the
authoritative source document.

**PW-5.** **T2 — Moderate.** Entities from reliable secondary sources
(e.g., Wikidata, curated knowledge bases, vendor white papers, enrichment from
cross-referencing multiple T1 sources). T2 entities are committed to the main
graph but flagged for periodic spot-check review.

**PW-6.** T2 entities SHOULD include a `source_url` attribute. When the source is
another entity in the graph (e.g., enrichment derived from cross-referencing),
a `references` relation to the source entity MAY substitute for `source_url`.

**PW-7.** **T3 — Draft.** Entities with low confidence or generated by automated
processes (e.g., AI-suggested relations, inferred categorizations, unverified
mappings). T3 entities MUST be isolated in the draft subgraph and MUST NOT be
committed to the main graph until a human reviewer approves them.

**PW-8.** T3 entities MUST have an associated ReviewItem entity created during
the review pass (see Section 7).

**PW-9.** **T4 — Dropped.** Entities that have been explicitly rejected by a
human reviewer or that have expired without approval (see Section 9). T4 is a
terminal state — entities do not transition out of T4.

**PW-10.** T4 entities MUST have their `confidence` attribute updated to `"dropped"`
and SHOULD be removed from the graph within one population run after being
marked.

### 3.2 Source Criteria

**PW-11.** Population tooling SHOULD document its source-to-tier mapping so that
reviewers can audit why an entity received a particular tier. The mapping
SHOULD be recorded in the project's documentation or `graph_card.json` ontology section.

**PW-12.** When a source could map to multiple tiers (e.g., a Wikidata entry
backed by an authoritative reference), the population tooling SHOULD assign the
higher-confidence tier and record the justification in a qualifier on the
`confidence` attribute.

## 4. Graph States

Entities in a DCG domain graph exist in one of four states. States are determined
by the entity's location (main graph vs. draft subgraph) and its `confidence`
attribute value.

### 4.1 State Definitions

**PW-13.** **Committed.** An entity in the main graph with `confidence`
value of `"official"` or `"moderate"`. Committed entities are fully linked (pass
strict validation if enabled) and are the authoritative knowledge in the graph.

**PW-14.** **Draft.** An entity in `graphs/draft.json` with `confidence` value
of `"draft"`. Draft entities are pending human review and MAY have incomplete
linking (exempt from strict mode's minimum entity-link requirement per
DCG-002 R-66).

**PW-15.** **Expired.** A draft entity whose associated ReviewItem has been
pending for longer than the project's TTL threshold (default: 30 days). Expired
entities are candidates for removal but require human confirmation.

**PW-16.** **Dropped.** An entity explicitly rejected by a human reviewer or
confirmed as expired. The entity's `confidence` attribute is set to `"dropped"` and
it is scheduled for removal.

### 4.2 State Transitions

**PW-17.** The following state transitions are valid:

```
                 ┌──────────────────────┐
                 │                      │
   ┌─────────┐  │  ┌──────────┐        │
   │  Draft  │──┴──▶ Committed │        │
   │  (T3)   │     │  (T1/T2) │        │
   └────┬────┘     └─────┬────┘        │
        │                │             │
        │  timeout       │  demote     │
        ▼                ▼             │
   ┌─────────┐     ┌──────────┐       │
   │ Expired │─────▶ Dropped  │───────┘
   │         │     │  (T4)    │  (re-draft)
   └─────────┘     └──────────┘
```

**PW-18.** `draft → committed`: A human reviewer approves the entity. The entity
is promoted from the draft subgraph to the main graph, its `confidence` attribute is
updated to `"official"` or `"moderate"`, and full linking attributes are added. The
entity MUST pass strict validation (if enabled) after promotion.

**PW-19.** `draft → expired`: The entity's ReviewItem has been pending beyond the
TTL threshold without action. The population run flags it; a human confirms
expiry.

**PW-20.** `committed → draft`: A committed entity is demoted for re-review
(e.g., source data changed, confidence downgraded). The entity is moved back to
the draft subgraph and a new ReviewItem is created.

**PW-21.** `expired → dropped`: A human confirms that the expired draft should be
removed.

**PW-22.** `draft → dropped`: A human reviewer explicitly rejects the entity.

**PW-23.** `dropped → draft`: In exceptional cases, a dropped entity MAY be
re-drafted if new evidence supports reconsideration. This MUST create a new
ReviewItem with a `review_reason` explaining the re-draft.

## 5. Population Run Protocol

A population run is a structured sequence of five ordered passes that adds,
updates, or validates entities and relations in a domain graph. Population runs
are idempotent — running the same pass with the same inputs SHOULD produce the
same graph state.

### 5.1 Pass Order

**PW-24.** A population run MUST execute passes in the following order:

1. Ontology pass
2. Domain pass
3. Entity pass
4. Relationship pass
5. Review pass

**PW-25.** Each pass SHOULD be completable independently. If a pass fails, the
population run SHOULD halt and report which pass failed and why, rather than
continuing to subsequent passes.

### 5.2 Ontology Pass

The ontology pass ensures the graph's type system is correctly configured before
any entities are created.

**PW-26.** The ontology pass MUST verify that the project's ontology
declarations (in `graph_card.json`) are loaded into the `OntologyStore`.

**PW-27.** The ontology pass MUST verify that all entity type types required for
the current population run are registered (either built-in or via
the `graph_card.json` ontology key).

**PW-28.** The ontology pass MUST verify that all properties required for the
current population run are registered.

**PW-29.** If the graph operates in strict mode (DCG-002 R-64), the ontology
pass MUST verify that strict mode is enabled and that all declarations are
in place before proceeding.

**PW-30.** The ontology pass SHOULD report any type or property aliases that
should be migrated to canonical names.

### 5.3 Domain Pass

The domain pass establishes the domain structure — top-level domains, sub-domains,
and the `part of` hierarchy between them.

**PW-31.** The domain pass MUST create or verify the existence of all Domain
entities required for the current population run.

**PW-32.** Domain entities MUST have `instance of → dcg:meta:Domain` attributes
(DCG-002 R-39).

**PW-33.** Sub-domain entities MUST have `part of` attributes linking them to their
parent domain, forming a hierarchy (DCG-002 R-51).

**PW-34.** Domain entities are always T1 (official) — they define the graph's
structure and do not require human review.

**PW-35.** The domain pass SHOULD verify that the number of top-level domains
is within the recommended range (1–10 per DCG-002 R-52).

### 5.4 Entity Pass

The entity pass creates or updates the substantive entities in the graph —
the items that represent real-world concepts, artifacts, or classifications.

**PW-36.** Every entity created in the entity pass MUST have at minimum:
- An `instance of` attribute referencing a registered entity type type
- A `confidence` attribute with the appropriate tier value
- A `part of` attribute (for T1 and T2 entities committed to the main graph)

**PW-37.** T1 and T2 entities MUST be added to the main graph store
(`graphs/default.json`) using `add_entity()`.

**PW-38.** T3 entities MUST be added to the draft subgraph store
(`graphs/draft.json`). They MUST NOT be added to the main graph store.

**PW-39.** T3 entities MAY omit the `part of` attribute at creation time. The
`part of` attribute MUST be added during promotion (draft → committed transition).

**PW-40.** The entity pass MUST be idempotent. Adding an entity with a UID that
already exists MUST update the existing entity (DCG-002 R-72), not create a
duplicate.

**PW-41.** Entities SHOULD include domain-specific identifier attributes
(e.g., `cwe_id`, `wikidata_id`, `owasp_id`) when the source data provides them.

### 5.5 Relationship Pass

The relation pass connects entities to each other via typed relations.

**PW-42.** Relations MUST use registered relation names. In strict mode,
unregistered relations are rejected (DCG-002 R-65).

**PW-43.** Cross-reference relations (e.g., `same as` between a CWE entry and
its Wikidata counterpart) SHOULD be created when external identifier attributes allow
matching.

**PW-44.** The relation pass SHOULD verify that both source and target
entities exist before creating a relation. If the target is a T3 draft entity
and the source is T1/T2, the relation SHOULD be deferred until promotion.

**PW-45.** Relations between entities in different confidence tiers SHOULD be
documented with a qualifier indicating the cross-tier nature, so reviewers can
assess whether the lower-confidence side warrants promotion.

### 5.6 Review Pass

The review pass creates ReviewItem entities for all entities that require human
review.

**PW-46.** The review pass MUST create a ReviewItem entity for every T3 (draft)
entity that does not already have an associated ReviewItem.

**PW-47.** The review pass SHOULD create ReviewItem entities for a sample of T2
(moderate) entities for spot-check verification. The sampling strategy is
implementation-defined but SHOULD be documented.

**PW-48.** ReviewItem creation follows the convention defined in Section 7.

## 6. Draft Subgraph Convention

T3 (draft) entities are isolated in a separate subgraph file to prevent
unreviewed content from polluting the main graph.

**PW-49.** Draft entities MUST be stored in a subgraph file at the path
`graphs/draft.json` relative to the project root.

**PW-50.** The project's graph index (`graphs` array in `graph_card.json`) MUST
include an entry for the draft subgraph:
```json
{"id": "draft", "file": "graphs/draft.json"}
```

**PW-51.** The draft subgraph MUST be created via `DcgProject.save_graph("draft",
"graphs/draft.json")` or equivalent API (DCG-002 R-104).

**PW-52.** Draft entities are exempt from strict mode's minimum entity-link
requirement (DCG-002 R-66). A draft entity MAY have only an `instance of` attribute
without a `part of` attribute.

**PW-53.** The draft subgraph store MUST inherit the parent project's ontology
via `OntologyStore.merge()` (DCG-002 R-60) so that draft entities are validated
against the same type system.

**PW-54.** On promotion (draft → committed), the implementation MUST:
1. Add the entity to the main graph store with full linking attributes
2. Verify the entity passes strict validation (if enabled)
3. Remove the entity from the draft subgraph store
4. Update the entity's `confidence` attribute to `"official"` or `"moderate"`
5. Remove the associated ReviewItem

**PW-55.** On rejection (draft → dropped), the implementation MUST:
1. Update the entity's `confidence` attribute to `"dropped"` in the draft subgraph
2. Record the rejection reason on the associated ReviewItem
3. Schedule the entity for removal in the next cleanup cycle

**PW-56.** The draft subgraph SHOULD be saved after each promotion or rejection
to persist the state change immediately.

## 7. ReviewItem Convention

ReviewItem entities are the mechanism for tracking what needs human attention.
They connect the automated population process to human reviewers.

**PW-57.** A ReviewItem MUST be a standard DCG entity with:
- `instance of → dcg:meta:ReviewItem`
- A `review_reason` attribute (string) explaining why the target entity needs review
- A `review_action` attribute (string) with one of: `verify`, `merge`, `split`,
  `remove`, `reclassify`

**PW-58.** A ReviewItem MUST be linked to its target entity via a `references`
relation: `ReviewItem --references--> TargetEntity`.

**PW-59.** ReviewItem entities are themselves T1 (official) — they are
structural artifacts of the population process, not domain knowledge requiring
review.

**PW-60.** ReviewItem entities MUST be stored in the same graph store as their
target entity. ReviewItems for draft entities go in the draft subgraph;
ReviewItems for committed entities (T2 spot-checks) go in the main graph.

**PW-61.** The `review_action` attribute values carry the following semantics:

| Action | Meaning |
|--------|---------|
| `verify` | Confirm the entity's data is correct. Promote if draft, keep if committed. |
| `merge` | Entity duplicates an existing entity. Merge and redirect (DCG-002 R-86). |
| `split` | Entity conflates multiple concepts. Split into separate entities. |
| `remove` | Entity is incorrect or irrelevant. Drop it. |
| `reclassify` | Entity's type or domain membership is wrong. Fix classification. |

**PW-62.** ReviewItem entities SHOULD include additional context attributes when
available:
- `source_url` — link to evidence supporting the review action
- `confidence` — the AI's confidence in the suggested action (as a percentage
  string, e.g., `"85%"`)

**PW-63.** After a human resolves a ReviewItem (by performing the action), the
ReviewItem MUST be removed from the graph. ReviewItems are transient — they
exist only while the review is pending.

**PW-64.** A single entity MAY have multiple ReviewItems if different aspects
require independent review (e.g., one for type classification, another for a
specific relation).

## 8. Human Review Protocol

Human review is the quality gate between automated population and authoritative
knowledge. This section defines the workflow for reviewers.

### 8.1 Discovering Pending Reviews

**PW-65.** Reviewers SHOULD use MCP query tools to discover pending ReviewItems:
- `dcg_query_entities(instance_of="ReviewItem")` — list all ReviewItems in the
  active graph
- `dcg_stack_query(entity_type="ReviewItem")` — list ReviewItems across all
  stack layers

**PW-66.** For each ReviewItem, reviewers SHOULD use `dcg_query_by_name` on the
target entity (via the `references` relation) to see the entity's full context:
attributes, relations, and position in the domain hierarchy.

### 8.2 Review Actions

**PW-67.** For each ReviewItem, the reviewer MUST perform one of the following
actions:

- **Approve (verify)**: Confirm the entity is correct. For draft entities, this
  triggers promotion (PW-54). For committed entities (spot-check), the
  ReviewItem is simply removed.

- **Reject (remove)**: The entity is incorrect. For draft entities, this
  triggers rejection (PW-55). For committed entities, this triggers demotion
  to dropped state with a new ReviewItem recording why.

- **Merge**: The entity duplicates an existing entity. The reviewer identifies
  the merge target, the implementation creates a `redirected to` attribute on the
  duplicate (DCG-002 R-86), and the duplicate is retracted.

- **Split**: The entity conflates multiple concepts. The reviewer identifies
  the split, the implementation creates new entities for each concept, migrates
  relevant attributes, and retracts the original.

- **Reclassify**: The entity's `instance of` or `part of` attributes are wrong.
  The reviewer specifies the correct classification, the implementation updates
  the attributes.

**PW-68.** After performing a review action, the reviewer (or automated tooling)
MUST:
1. Execute the action on the target entity
2. Remove the resolved ReviewItem from the graph
3. If the action changed the entity's confidence tier, update the `confidence`
   attribute accordingly
4. Save the affected graph store(s)

### 8.3 Batch Review

**PW-69.** Implementations SHOULD support batch review workflows where multiple
ReviewItems with the same `review_action` can be processed together (e.g.,
approve all T3 entities from a trusted secondary source after spot-checking
a sample).

**PW-70.** Batch operations MUST still validate each entity individually —
batch approval MUST NOT bypass strict validation on promotion.

## 9. Expiry and Cleanup

Draft entities that linger without review become stale. This section defines
expiry rules and cleanup procedures.

**PW-71.** Draft entities whose associated ReviewItem has been pending for longer
than the project's TTL threshold are candidates for expiry. The default TTL
threshold is 30 days.

**PW-72.** Expiry is NOT automatic. A population run SHOULD flag expired
ReviewItems (e.g., by updating `review_action` to `"remove"` with a
`review_reason` noting the expiry), but a human MUST confirm the removal.

**PW-73.** Dropped entities (T4) SHOULD be retained in the draft subgraph for
one additional population run after being marked, to provide an audit trail.
After that retention period, they SHOULD be removed.

**PW-74.** After cleanup (removal of dropped entities), implementations SHOULD
run `purge_retracted()` (DCG-002 R-79) to permanently remove retracted entities and relations.

## 10. Layer and Stack Conventions

Population runs interact with DCG's composition model (DCG-002 Section 7.5).
This section defines conventions for how population data flows through
composition layers.

**PW-75.** A base knowledge layer SHOULD contain T1 and T2 entities from
authoritative sources. This layer represents the canonical domain knowledge.

**PW-76.** Extension layers SHOULD contain T2 entities from secondary sources
that enrich the base layer. Extension layers declare `extends` referencing
the base layer in a stack manifest.

**PW-77.** A draft layer (containing the draft subgraph) is NOT a versioning
layer — it is a separate graph file within the same project. It SHOULD NOT
appear as a composition layer in a stack manifest. Draft entities are project-
internal until promoted.

**PW-78.** Code and runtime layers (populated by automated tools like code
parsers) SHOULD extend the product or base layer in a stack via `extends`.

---

## 11. Strict Mode Integration

Population runs interact with DCG's strict mode (DCG-002 Section 6.5.4) to
ensure type safety and structural integrity.

**PW-79.** Population runs SHOULD target strict-mode graphs. Strict mode
catches ontology violations (unregistered types, properties, relations)
at write time rather than letting them accumulate.

**PW-80.** The draft subgraph MAY operate in non-strict mode (relaxed minimum
entity-link count) to accommodate incomplete T3 entities that lack `part of`
attributes. The `DcgProject` implementation handles this by allowing draft entities
with only `instance of` attributes.

**PW-81.** On promotion from draft to main graph, the entity MUST pass strict
validation. If the entity fails validation (e.g., missing required attributes or
unregistered types), promotion MUST be rejected and the ReviewItem MUST be
updated with the validation error as the `review_reason`.

## Appendix A: Population Run Checklist

A quick-reference checklist for operators running a population pass:

1. **Ontology pass**
   - [ ] `graph_card.json` ontology declarations loaded
   - [ ] All required types registered
   - [ ] All required properties registered
   - [ ] Strict mode enabled (if applicable)
   - [ ] Alias migrations identified

2. **Domain pass**
   - [ ] Top-level domain(s) exist
   - [ ] Sub-domains created with `part of` hierarchy
   - [ ] Domain count within recommended range (1–10 top-level)

3. **Entity pass**
   - [ ] T1/T2 entities in main graph with full attributes
   - [ ] T3 entities in `graphs/draft.json`
   - [ ] All entities have `instance of` + `confidence` attributes
   - [ ] T1/T2 entities have `part of` attributes
   - [ ] Domain-specific identifier attributes set (cwe_id, wikidata_id, etc.)

4. **Relationship pass**
   - [ ] Relations use registered relation names
   - [ ] Cross-references created (same as, references)
   - [ ] No dangling references to nonexistent entities

5. **Review pass**
   - [ ] ReviewItem for every T3 entity
   - [ ] Spot-check ReviewItems for T2 sample
   - [ ] ReviewItems linked to targets via `references`
   - [ ] Each ReviewItem has `review_reason` + `review_action`

6. **Post-run**
   - [ ] All graph stores saved
   - [ ] Expired drafts flagged
   - [ ] Previous-run dropped entities cleaned up
   - [ ] Run `purge_retracted()` to clean up retracted data
